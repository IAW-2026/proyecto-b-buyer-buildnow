Buyer App · guía interna

Estructura del proyecto

Esta guía resume qué responsabilidad tiene cada carpeta importante y cómo viaja una acción de usuario desde la interfaz hasta las integraciones, Prisma y la respuesta final.

Stack principal

Next.js App Router, React, TypeScript, Clerk, Prisma, Tailwind y clientes HTTP para servicios externos o mocks locales.

Más detalle

La app combina server components para lectura inicial, componentes cliente para interacción y server actions/API routes para ejecutar operaciones protegidas en el servidor.

Mapa rápido

Carpetas y archivos clave

src/app

Rutas, layouts y API routes

Define las pantallas del App Router, grupos como (public), (private), (onboarding) y endpoints internos bajo api.

Más detalle
  • page.tsx representa una pantalla.
  • layout.tsx envuelve rutas con estructura común.
  • api/*/route.ts expone endpoints internos, como IA o georef.
src/components

Interfaz reutilizable

Contiene piezas visuales por dominio: carrito, productos, tiendas, órdenes, direcciones, admin, búsqueda y asistente IA.

Más detalle

Acá vive la experiencia visible. Algunos componentes son server components y otros usan estado/eventos del navegador, como formularios, sidebar del carrito o asistente IA.

src/actions

Server actions

Expone operaciones invocables desde componentes, valida sesión o rol y delega la lógica real a servicios.

Más detalle

Funcionan como frontera entre UI y servidor. Devuelven respuestas simples para que la interfaz pueda mostrar éxito, errores de permisos o validaciones de negocio.

src/server/services

Casos de uso

Coordina reglas de negocio. Por ejemplo, catálogo, carrito, compradores, admin y recomendaciones IA.

Más detalle

Es la capa que decide qué hacer. Puede combinar repositorios, integraciones externas y validaciones como stock suficiente o carrito con productos de una sola tienda.

src/server/repositories

Acceso a base de datos

Encapsula consultas Prisma para buyers, direcciones, carritos y datos administrativos.

Más detalle

No debería contener lógica visual ni saber de componentes. Su trabajo es leer, crear, actualizar o borrar entidades del modelo local de Buyer.

src/server/integrations

Servicios externos

Clientes server-only para Seller, Delivery y Gemini. Separan el contrato externo de la lógica de la app.

Más detalle

Centraliza URLs, payloads y respuestas de otros sistemas. Por ejemplo, Seller entrega tiendas/productos y Gemini genera recomendaciones del asistente.

src/lib

Infraestructura compartida

Helpers de autenticación, cliente Prisma, clientes HTTP base, rate limit, serialización y validaciones.

Más detalle

Es código transversal: piezas usadas por varias capas. Buenos candidatos son utilidades pequeñas, clientes base y funciones de seguridad o normalización.

src/context

Estado del cliente

Guarda estado React compartido, como el carrito visible en la experiencia de compra.

Más detalle

Conviene usarlo para estado de UI que varios componentes necesitan leer o actualizar sin pasarse props por muchas capas.

prisma

Modelo persistente

Incluye schema.prisma y migraciones de la base de datos local de Buyer.

Más detalle

El schema define entidades como buyer, direcciones, carrito e items. Las migraciones dejan registrado cómo evolucionó la base.

public

Assets estáticos

Logo, favicon e imágenes mock de productos servidas directamente por Next.

Más detalle

Todo lo que está acá se puede servir como archivo estático. Sirve para imágenes de producto, íconos y recursos que no necesitan procesamiento del servidor.

scripts

Automatizaciones locales

Scripts auxiliares, como la generación de imágenes mock para el catálogo.

Más detalle

Son tareas de apoyo para desarrollo. No forman parte directa del runtime de Next, pero ayudan a preparar datos o assets.

docs

Documentación del proyecto

Documentos HTML o Markdown para explicar arquitectura, endpoints y decisiones del sistema.

Más detalle

Es el lugar para guías de lectura rápida: estructura, contratos consumidos, decisiones técnicas y flujos principales.

Cómo pensar la app

Separación por capas

  • Las pantallas y componentes no deberían hablar directo con Prisma ni con APIs externas. Usan server actions, API routes o servicios.
  • Los servicios concentran reglas: stock, tienda única en carrito, búsqueda, recomendaciones, armado de órdenes o datos de tracking.
  • Las integraciones aíslan contratos externos. Si no hay URL real de Seller, el cliente usa el mock de src/server/mockSeller.
Más detalle

Esta separación ayuda a cambiar una capa sin reescribir todo: se puede reemplazar un mock por una API real, ajustar una regla de negocio en services o modificar UI sin tocar persistencia.

Ejemplo de lectura: src/app/[storeId]/page.tsx arma la página de una tienda llamando a getAllStores, getCatalogCategories y getStoreProductsService. Esos servicios consultan la integración Seller y devuelven datos listos para renderizar.

Ejemplo de escritura: agregar al carrito pasa por addToCartAction, valida el buyer con Clerk, ejecuta addProductToCart y persiste el item con Prisma.

Más detalle

En lectura, el server component puede esperar datos antes de renderizar. En escritura, la acción vuelve con un resultado controlado para que el componente cliente actualice el estado o muestre un mensaje.

Flujo de ejemplo

Agregar un producto al carrito

UI

ProductCard

El usuario toca agregar o cambia la cantidad del producto.

Más detalle

El componente captura el evento y llama a una acción del carrito.

Action

cart.actions.ts

addToCartAction(productId) recibe la intención.

Más detalle

La action traduce errores técnicos a respuestas útiles para la UI.

Auth

requireBuyer

Confirma sesión de Clerk y permisos de comprador.

Más detalle

Si no hay sesión o rol correcto, corta el flujo antes de tocar datos.

Service

buyer.service.ts

Busca o crea carrito, valida stock y evita mezclar tiendas.

Más detalle

Acá aparecen reglas como DIFFERENT_STORE_CART o INSUFFICIENT_STOCK.

Data

Seller + Prisma

Seller aporta detalle del producto; Prisma guarda el carrito.

Más detalle

Seller puede venir de API real o mock; Prisma persiste la parte propia de Buyer.

Response

UI actualizada

La action devuelve éxito o error y el componente refresca estado.

Más detalle

El usuario ve la cantidad nueva, el carrito actualizado o un mensaje de error.

Regla práctica: si una carpeta empieza con server, su código pertenece al lado servidor. Si está en components o context, pensarlo como experiencia de usuario. Si está en lib, debería ser infraestructura reusable y sin lógica de pantalla.
